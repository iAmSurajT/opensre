title: "OpenClaw"
description: "Connect OpenSRE to OpenClaw with the minimum setup needed to run investigations from your editor"
---

OpenClaw can call OpenSRE's `run_rca` MCP tool so you can investigate alerts from your editor.

## Setup

### 1. Start OpenSRE MCP

```bash
opensre-mcp
```

### 2. Add OpenSRE to OpenClaw

In OpenClaw, add this MCP server:

```json
{
    "mcpServers": {
        "opensre": {
            "command": "opensre-mcp",
            "args": []
        }
    }
}
```

If `opensre-mcp` is not on your `PATH`, run it through `uv`:

```json
{
    "mcpServers": {
        "opensre": {
            "command": "uv",
            "args": ["run", "opensre-mcp"]
        }
    }
}
```

### 3. Configure one observability integration

Run the wizard once and connect Datadog, Grafana, Sentry, or another backend:

```bash
opensre integrations setup
```

### 4. Run a smoke test

Run the fixture directly with the investigate command:

```bash
opensre investigate -i tests/fixtures/openclaw_test_alert.json
```

Once that works, you can call OpenSRE from OpenClaw through the `run_rca` MCP tool.

## Verify before investigating

Before debugging issues, quickly validate your setup:

- Run `opensre integrations verify openclaw`
- Ensure the bridge command is `openclaw mcp serve`
- Confirm the OpenClaw Gateway is running when using the stdio bridge

If any of these checks fail, fix them first before investigating further.

## Optional: let OpenSRE call OpenClaw

If you want the OpenSRE investigation pipeline to query OpenClaw during RCA runs, use the OpenClaw stdio bridge:

```bash
export OPENCLAW_MCP_MODE=stdio
export OPENCLAW_MCP_COMMAND=openclaw
export OPENCLAW_MCP_ARGS="mcp serve"
```

Keep the OpenClaw Gateway running while you investigate:

```bash
openclaw gateway run
```

Verify it with:

```bash
opensre integrations verify openclaw
```

## Troubleshooting

- `opensre-mcp` not found: run it as `uv run opensre-mcp`.
- `run_rca` says no integrations configured: run `opensre integrations setup` and add Datadog, Grafana, or Sentry.
- Verify fails against `http://127.0.0.1:18789/`: that URL is the OpenClaw Control UI/Gateway, not the MCP bridge. Use `openclaw mcp serve`.
- `search_openclaw_conversations` fails with `Connection closed` or `ECONNREFUSED`: the OpenClaw Gateway is not running. Check `openclaw gateway status`, then run `openclaw gateway run`.
- OpenClaw CLI prints `Node.js v22.12+ is required`: switch your shell to Node 22. With `nvm`: `nvm install 22 && nvm use 22 && nvm alias default 22`. With Homebrew: `brew install node@22 && brew link --overwrite node@22`. Then re-run `openclaw --version`.

## End-to-end tests

The suite at `tests/e2e/openclaw/` boots a local Gateway, injects a fault, and asserts OpenSRE identifies the failure mode.

Prerequisites:

- `openclaw` CLI on `$PATH` (tests skip when absent).
- Node `22.12+` active in the current shell.
- Optional: `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GEMINI_API_KEY` for the full-RCA sub-tests; without one, only the use-case sub-tests run.

Run the suite:

```bash
make test-openclaw
```

Run one scenario:

```bash
uv run python -m pytest -m e2e -v tests/e2e/openclaw/test_gateway_down.py
```

Scenarios:

- `test_gateway_down.py` — bridge runs, Gateway doesn't; RCA must recommend `openclaw gateway run`.
- `test_wrong_endpoint.py` — config points at the Control UI port (`18789`); RCA must recommend `stdio`.
- `test_tool_call_timeout.py` — a fixture MCP tool sleeps past `OpenClawConfig.timeout_seconds`; RCA must surface "timeout".

CI: `.github/workflows/e2e-openclaw.yml` runs the suite on PRs labeled `ci:openclaw`. Add `ANTHROPIC_API_KEY` to repo secrets to enable the full-RCA sub-tests.

For the deeper reference (layout, fault-injection contract, how to add a new scenario) see [`tests/e2e/openclaw/README.md`](https://github.com/Tracer-Cloud/opensre/tree/main/tests/e2e/openclaw/README.md).
